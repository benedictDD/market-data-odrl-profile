The set of patterns provided here show how the terms defined in the Market Data Profile for ODRL should be combined to model the permissions, prohibitions, obligations, and constraints specified in the licences controlling the use of market data as ODRL policies.

The core intended audience for this document are the authors of ODRL policies and the implementors of systems interpreting them. The objective is to match the former's intentions with the latter's expectations.

Patterns

Duties

A Duty is defined in ODRL as the obligation to exercise an Action.

The Party exercising the action is the Subject of the Duty. The Party affected is the Object.

If the specified action must be exercised more than once to fulfill the Duty, then the count constraint should be used to indicate the number of times the action must be taken.

A Duty may also specify a target to indicate the Entity, Activity, or Party to which the action applies.

Some duties can be activated by events. If so, the duty should specify its activation condition in the form of a constraint that when satisfied, activates the Duty.

Some duties may be activated by the passage of time. If so, the duty should specify its activation condition in the form of a time interval constraint (e.g. monthly) that when satisfied, activates the Duty.

Just eight types of Duty capture most of the obligations found in market data licenses: Attachments, Requests and Consents, Notifications, Invoices and Payments, Explicit Agreements, and Reporting.

Pattern 1.1 - Attachments

Market data licenses can require Attributions, Disclaimers, and Proscriptions to be attached to data that is transferred from one Party to another.

The wording of these can be specified in the policy using the wording property of the Attribution.

Attribution Duty

Attribution duties require the Subject to attach a declaration of the IP Owner's title to the Asset.

The duty is usually activated when the Asset is transferred to a third-party, i.e. the recipient constraint is triggered by an External Party.

Definition:Requires and specifies an Attribution to be attached to the data whenever the recipient is an External Party.
Subject:The Party that attaches the Attribution - usually the Party making the transfer.
Object:The Party that receives the Attribution - usually the recipient of the transfer.
Action:md:Attach
Target:md:Attribution
Activation:When an external Party becomes a Recipient of the Asset controlled by the parent Permission.

Attribution Code

                        []   a             odrl:Duty ;
                             md:subject    <http://www.example.org/transferer> ;
                             md:object     <http://www.example.org/recipient> ;
                             md:recipients [ a md:ExternalParty ] ;
                             odrl:action   [
                                                a               md:Attach ;
                                           ] ;
                             odrl:target   [
                                                a               md:Attribution ;
                                                md:wording      "The market data is the property of Chicago Mercantile Exchange Inc. or itâ€™s licensors as applicable. All rights reserved, or otherwise licensed by Chicago Mercantile Exchange Inc."
                                           ] .

Disclaimer Duty

Disclaimers must sometimes be attached to derived data to clarify (usually distance) the relationship between the IP Owner of the source Asset and the derived Asset.

Disclaimer duties follow the same pattern as Attribution Duties except the value of the target property is now an Disclaimer

Proscription Duty

Proscriptions are written instructions on the use of an Asset that must sometimes be attached the Asset before its distribution.

When the instructions are legally binding, and must be formally agreed with the recipient, use the Agreement Duty pattern below instead of this one.

Proscription duties follow the same pattern as Attribution Duties except the value of the target property is now an Proscription

Pattern 1.2 - Agreements

Market data licenses can require Proscriptions to be legally binding before an Asset can be transferred.

The wording of these can be specified in the policy using the wording property of the Proscription.

Agreement Duty

Agreement duties require the Subject to come to a legally binding agreement with the recipient of the Asset that enforces the Proscription.

Definition:Requires and specifies a Proscription to be agreed in a legally binding manner with the recipient whenever the recipient is an External Party.
Subject:The Party that seeks the Agreement - usually the Pary making the tranfer.
Object:The Party that accedes to the Agreement - usually the recipient of the transfer.
Action:md:Agree
Target:md:Proscription
Activation:When an external Party becomes a Recipient of the Asset controlled by the parent Permission.

Agreement Code

                        []   a             odrl:Duty ;
                             md:subject    <http://www.example.org/transferer> ;
                             md:object     <http://www.example.org/recipient> ;
                             md:recipients [ a md:ExternalParty ] ;
                             odrl:action   [
                                                a               md:Agree ;
                                           ] ;
                             odrl:target   [
                                                a               md:Proscription ;
                                                md:wording      "..."
                                           ] .

Pattern 1.3 - Notifications

Data Originators and Providers sometimes require notification of particular events or changes in the state-of-the-world.

The type of change is specified using the target property. The most common one is when the Asset is accessed by an External Party (e.g. a Service Facilitator).

Sometimes, it's the Consumer that receives the notification. For example, notification of a reasonable suspicion of data misuse can trigger an Audit.

Usage Notification Duty

Usage notification requires the Subject to notify the Object that they have started to use the Asset.

The duty is activated when the users constraint is satisfied.

Definition:Requires a notification be generated when an Internal Party starts to use an Asset.
Subject:The Party that provides the Notification - usually the Party playing the role of data Consumer.
Object:The Party that receives the Notification - usually the Party playing the role of data Provider or Originator.
Action:md:Notify
Target:md:Usage
Activation:When users are associated with the parent Permission.

Usage Notification Code

                        []   a             odrl:Duty ;
                             md:subject    <http://www.example.org/consumer> ;
                             md:object     <http://www.example.org/provider> ;
                             md:users      [ a md:InternalParty ] ;
                             odrl:action   [
                                                a               md:Notify ;
                                           ] ;
                             odrl:target   [
                                                a               md:Usage ;
                                           ] .

External-Party Notification Duty

External-Party notification requires the Subject to notify the Object that an external party (usually a Service Facilitator) has access to the Asset and to identify them.

The duty is activated when the recipients constraint is satisfied.

External-Party Notification Code

                        []   a             odrl:Duty ;
                             md:subject    <http://www.example.org/transferer> ;
                             md:object     <http://www.example.org/provider> ;
                             md:recipients [
                                                a               md:ExternalParty ;
                                                md:role         [ a md:ServiceFacilitator ] ;
                                           ] ;
                             odrl:action   [
                                                a               md:Notify ;
                                           ] ;
                             odrl:target   [
                                                a               md:ExternalParty ;
                                                md:role         [ a md:ServiceFacilitator ] ;
                                           ] .

There are several data use cases that require that Consent be given before they can be satisfied. Often consent is required for distribution to a third-party or for derivations. In these cases, the data Originator must provide the consent.

But there are also cases - like an Audit - where the consent of the Data Consumer is required.

Pattern 1.4 - Requesting Consent

Consent Duty

Segmenting Assets

Many rules provide access to only a part of an Asset. These segmentations can be by the type of the content (e.g. access to just French equities or only the index values, not the constituents and weightings) or by temporal reach (e.g. access to only three years of historical data). The first we call a content segmentation and the second, a temporal segmentation.

Temporal Segmentations

The most common temporal segmentations control access to historical data or time series data.

The interval itself must be defined using the OWL Time Ontology. Often this is effected using its has XSD Duration property.

                []   odrl:timeInterval     [
                                               a                    time:ProperInterval ;
                                               time:hasXSDDuration  "P1M"^^xsd:duration
                                           ] .

Content Segmentations

Not all Duties need be fulfilled the moment they activate. Often a grace period is provided in which the Duty can be fulfilled. These are specified using the deadline delta property. It defines an interval (frequently 30 days) within which the containing Duty must be fulfilled.

                []   md:hasDeadlineDelta  [
                                               a                    time:ProperInterval ;
                                               time:hasXSDDuration  "P30D"^^xsd:duration
                                           ] .

Time

Many activities required by Duties in market data licenses are activated by the clock or calendar. For example, reporting may need to be done monthly, 30 days in arrears. Payment schedules are often similar.

Periodicity

If a duty is activated on a recurring interval of time (e.g. monthly or every 14 days), then the time interval property from the ODRL Common Vocabulary must be used to specify the interval.

The interval itself must be defined using the OWL Time Ontology. Often this is effected using its has XSD Duration property.

                []   odrl:timeInterval     [
                                               a                    time:ProperInterval ;
                                               time:hasXSDDuration  "P1M"^^xsd:duration
                                           ] .

Deadlines

Not all Duties need be fulfilled the moment they activate. Often a grace period is provided in which the Duty can be fulfilled. These are specified using the deadline delta property. It defines an interval (frequently 30 days) within which the containing Duty must be fulfilled.

                []   md:hasDeadlineDelta  [
                                               a                    time:ProperInterval ;
                                               time:hasXSDDuration  "P30D"^^xsd:duration
                                           ] .

Temporal Constraints

Temporal constaints specify access to historical data using the temporal property. The interval allowing access to the latest three years of historical data would look so:

                []   md:temporal  [
                                               a                    time:ProperInterval ;
                                               time:hasXSDDuration  "P3Y"^^xsd:duration
                                           ] .

Specified Intervals

Sometimes, for example, when an effective interbal is being specified, we need to indicate interval than run from a specified data and/or to one. We rely on some properties from the Time Ontology to do so.

                []   tpl:effectiveInterval  [
                                               a                    time:ProperInterval ;
                                               time:hasBeginning  [
                                                    time:time:inXSDDate  "2020-01-01T00:00:00Z"^^xsd:dateTime
                                                    ] .  
                                               time:hasEnd        [
                                                    time:time:inXSDDate  "2022-01-01T00:00:00Z"^^xsd:dateTime
                                                    ] .     
                                           ] .
